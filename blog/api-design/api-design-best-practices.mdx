## üèóÔ∏è Understanding API Design {#understanding-api-design}

API design is about creating interfaces that are:

- **Intuitive** - Easy to understand and use
- **Consistent** - Follows predictable patterns
- **Scalable** - Can grow with your application
- **Maintainable** - Easy to modify and extend
- **Secure** - Protects data and prevents abuse

### **Why Good API Design Matters**

```http
# Bad API Design Example
GET /getUserData?userId=123&includePosts=true&includeComments=false

# Good API Design Example
GET /api/v1/users/123?include=posts&exclude=comments
```

## üéØ RESTful Principles {#restful-principles}

REST (Representational State Transfer) provides a set of architectural
principles for designing networked applications.

### **Core REST Principles**

#### **1. Stateless Communication**

Each request contains all information needed to process it.

```http
# ‚úÖ Stateless - Each request is self-contained
POST /api/v1/users
Content-Type: application/json

{
  "name": "John Doe",
  "email": "john@example.com"
}

# ‚ùå State-dependent - Requires previous context
POST /api/v1/create-user-step-2
Content-Type: application/json

{
  "userId": "temp_123"
}
```

#### **2. Client-Server Architecture**

Clear separation between client and server responsibilities.

#### **3. Cacheable Responses**

Responses should be cacheable when appropriate.

```http
# Cacheable response
GET /api/v1/products
Cache-Control: public, max-age=300

# Non-cacheable response
POST /api/v1/orders
Cache-Control: no-cache
```

#### **4. Uniform Interface**

Consistent interface across all endpoints.

#### **5. Layered System**

Ability to add layers (load balancers, proxies) without affecting clients.

## üè∑Ô∏è Resource Naming Conventions {#resource-naming-conventions}

### **URL Structure Best Practices**

```http
# ‚úÖ Good resource naming
GET /api/v1/users
GET /api/v1/users/123
GET /api/v1/users/123/posts
GET /api/v1/products/categories/electronics

# ‚ùå Avoid these
GET /api/v1/getAllUsers
GET /api/v1/userData?id=123
GET /api/v1/user_posts?user_id=123
```

### **Naming Guidelines**

- **Use nouns, not verbs** - Resources should be things, not actions
- **Use plural nouns** - `/users`, `/products`, `/orders`
- **Use lowercase** - `/api/v1/users`, not `/api/v1/Users`
- **Use hyphens for multi-word resources** - `/user-profiles`, `/order-items`
- **Be consistent** - Use the same patterns throughout your API

### **Nested Resources**

```http
# ‚úÖ Hierarchical relationships
GET /api/v1/users/123/posts
GET /api/v1/companies/456/employees
GET /api/v1/products/789/reviews

# ‚ùå Deep nesting (avoid more than 3 levels)
GET /api/v1/countries/india/states/karnataka/cities/bangalore/streets/mgroad/buildings
```

## üîÑ HTTP Methods & Status Codes {#http-methods--status-codes}

### **Standard HTTP Methods**

| Method   | Purpose                 | Safe | Idempotent |
| -------- | ----------------------- | ---- | ---------- |
| `GET`    | Retrieve data           | ‚úÖ   | ‚úÖ         |
| `POST`   | Create new resource     | ‚ùå   | ‚ùå         |
| `PUT`    | Update/replace resource | ‚ùå   | ‚úÖ         |
| `PATCH`  | Partial update          | ‚ùå   | ‚ùå         |
| `DELETE` | Remove resource         | ‚ùå   | ‚úÖ         |

### **HTTP Status Code Categories**

#### **2xx - Success**

```http
200 OK           # Request succeeded
201 Created      # Resource created successfully
204 No Content   # Success, no response body
```

#### **3xx - Redirection**

```http
301 Moved Permanently  # Resource moved permanently
302 Found             # Resource moved temporarily
304 Not Modified      # Resource not changed (caching)
```

#### **4xx - Client Errors**

```http
400 Bad Request       # Invalid request syntax
401 Unauthorized      # Authentication required
403 Forbidden         # Access denied
404 Not Found         # Resource doesn't exist
422 Unprocessable Entity # Validation failed
429 Too Many Requests # Rate limit exceeded
```

#### **5xx - Server Errors**

```http
500 Internal Server Error  # Unexpected server error
502 Bad Gateway           # Invalid response from upstream
503 Service Unavailable   # Server temporarily down
504 Gateway Timeout       # Upstream timeout
```

## üè∑Ô∏è API Versioning Strategies {#api-versioning-strategies}

### **Versioning Methods**

#### **1. URL Path Versioning** (Recommended)

```http
/api/v1/users
/api/v2/users
/api/v3/users
```

#### **2. Query Parameter Versioning**

```http
/api/users?version=1
/api/users?version=2
```

#### **3. Header Versioning**

```http
GET /api/users
Accept: application/vnd.api+json; version=1
```

#### **4. Content Negotiation**

```http
GET /api/users
Accept: application/vnd.company.api.v1+json
```

### **Versioning Best Practices**

```http
# ‚úÖ Semantic versioning
/api/v1.2/users    // Major.Minor
/api/v2023/users   // Date-based

# ‚ùå Avoid breaking changes in same version
# Version 1.0 should never break existing clients
```

## üîê Authentication & Authorization {#authentication--authorization}

### **Common Authentication Methods**

#### **1. API Keys**

```http
# Header-based
GET /api/users
X-API-Key: your-api-key-here

# Query parameter (less secure)
GET /api/users?api_key=your-key
```

#### **2. JWT (JSON Web Tokens)**

```http
# Authorization header
GET /api/users
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
```

#### **3. OAuth 2.0**

```http
# Access token
GET /api/protected-resource
Authorization: Bearer access_token_here
```

#### **4. Basic Authentication**

```http
GET /api/users
Authorization: Basic dXNlcjpwYXNzd29yZA==
```

### **Authorization Patterns**

```http
# ‚úÖ Role-based access
GET /api/admin/users    // Admin only
GET /api/users/profile  // Authenticated users
GET /api/public/posts   // Public access

# ‚úÖ Resource ownership
GET /api/users/123/posts // User's own posts
GET /api/users/456/posts // Forbidden (403)
```

## üß™ Testing & Validation {#testing--validation}

### **API Testing Strategy**

#### **1. Unit Tests**

```http
# Test individual API functions
describe('User API', () => {
  test('should create user successfully', async () => {
    const userData = { name: 'John', email: 'john@test.com' };
    const response = await request(app)
      .post('/api/users')
      .send(userData)
      .expect(201);

    expect(response.body.data).toHaveProperty('id');
    expect(response.body.data.name).toBe(userData.name);
  });
});
```

#### **2. Integration Tests**

```http
# Test complete request/response cycles
describe('User Registration Flow', () => {
  test('should complete full registration', async () => {
    # 1. Register user
    const registerResponse = await request(app)
      .post('/api/auth/register')
      .send({ email: 'john@test.com', password: 'password123' })
      .expect(201);

    # 2. Verify email
    const token = registerResponse.body.data.verificationToken;
    await request(app)
      .post('/api/auth/verify-email')
      .send({ token })
      .expect(200);

    # 3. Login
    const loginResponse = await request(app)
      .post('/api/auth/login')
      .send({ email: 'john@test.com', password: 'password123' })
      .expect(200);

    expect(loginResponse.body.data).toHaveProperty('accessToken');
  });
});
```

#### **3. Load Testing**

```http
# Test performance under load
import autocannon from 'autocannon';

const instance = autocannon({
  url: 'http://localhost:3000/api/users',
  connections: 100,
  duration: 10
});

instance.on('done', (result) => {
  console.log('Requests per second:', result.requests.average);
  console.log('Latency average:', result.latency.average);
});
```

## üîí Security Best Practices {#security-best-practices}

### **Input Validation & Sanitization**

```http
# ‚úÖ Validate and sanitize all inputs
const userSchema = Joi.object({
  name: Joi.string().min(2).max(50).trim().required(),
  email: Joi.string().email().lowercase().trim().required(),
  age: Joi.number().integer().min(13).max(120).optional()
});

app.post('/api/users', async (req, res) => {
  const { error, value } = userSchema.validate(req.body);
  if (error) {
    return res.status(422).json({
      error: 'Validation failed',
      details: error.details
    });
  }
  # Proceed with validated data
});
```

### **SQL Injection Prevention**

```javascript
// ‚ùå Vulnerable to SQL injection
const query = `SELECT * FROM users WHERE id = ${req.params.id}`;

// ‚úÖ Parameterized queries
const query = 'SELECT * FROM users WHERE id = ?';
const user = await db.query(query, [req.params.id]);

// ‚úÖ ORM protection
const user = await User.findByPk(req.params.id);
```

### **CORS Configuration**

```javascript
// ‚úÖ Proper CORS setup
const corsOptions = {
  origin: (origin, callback) => {
    const allowedOrigins = [
      'https://yourdomain.com',
      'https://app.yourdomain.com',
    ];
    if (allowedOrigins.includes(origin) || !origin) {
      callback(null, true);
    } else {
      callback(new Error('Not allowed by CORS'));
    }
  },
  credentials: true,
  methods: ['GET', 'POST', 'PUT', 'DELETE', 'OPTIONS'],
  allowedHeaders: ['Content-Type', 'Authorization'],
};

app.use(cors(corsOptions));
```

## ‚ö° Performance Optimization {#performance-optimization}

### **Caching Strategies**

```http
# ‚úÖ HTTP caching headers
app.get('/api/products', (req, res) => {
  res.set({
    'Cache-Control': 'public, max-age=300', // 5 minutes
    'ETag': generateETag(products),
    'Last-Modified': products.lastModified
  });
  res.json(products);
});

# ‚úÖ Redis caching for expensive operations
const cache = require('redis').createClient();

app.get('/api/analytics', async (req, res) => {
  const cacheKey = 'analytics_data';

  # Check cache first
  const cached = await cache.get(cacheKey);
  if (cached) {
    return res.json(JSON.parse(cached));
  }

  # Expensive operation
  const data = await calculateAnalytics();

  # Cache for 10 minutes
  await cache.setex(cacheKey, 600, JSON.stringify(data));

  res.json(data);
});
```

### **Database Optimization**

```http
# ‚úÖ Use indexes for frequently queried fields
db.collection('users').createIndex({ email: 1 });
db.collection('products').createIndex({ category: 1, price: 1 });

# ‚úÖ Pagination to prevent large result sets
app.get('/api/products', async (req, res) => {
  const page = parseInt(req.query.page) || 1;
  const limit = parseInt(req.query.limit) || 20;
  const skip = (page - 1) * limit;

  const products = await Product.find()
    .skip(skip)
    .limit(limit)
    .sort({ createdAt: -1 });

  const total = await Product.countDocuments();

  res.json({
    data: products,
    pagination: {
      page,
      limit,
      total,
      pages: Math.ceil(total / limit)
    }
  });
});
```

## üö´ Common Pitfalls to Avoid {#common-pitfalls-to-avoid}

### **1. Over-fetching/Under-fetching**

```http
# ‚ùå Over-fetching - getting too much data
GET /api/users (returns 50MB of user data when you only need names)

# ‚úÖ Selective fetching
GET /api/users?fields=name,email,id

# ‚ùå Under-fetching - too many requests (N+1 problem)
GET /api/users (returns users without posts)
GET /api/users/1/posts
GET /api/users/2/posts
# ... 100 more requests

# ‚úÖ Include related data
GET /api/users?include=posts
```

### **2. Inconsistent Naming**

```http
# ‚ùå Mixed conventions
GET /api/getUser
GET /api/create_user
GET /api/delete-user

# ‚úÖ Consistent naming
GET /api/users/{id}
POST /api/users
DELETE /api/users/{id}
```

### **3. Breaking Changes Without Versioning**

```http
# ‚ùå Breaking existing clients
# Version 1.0
{ "user_name": "John" }

# Version 1.1 (breaking change)
{ "fullName": "John" }

# ‚úÖ Proper versioning
GET /api/v1/users (old format)
GET /api/v2/users (new format)
```

### **4. Poor Error Messages**

```javascript
// ‚ùå Unhelpful errors
{ "error": "Internal server error" }

// ‚úÖ Helpful errors
{
  "error": "VALIDATION_ERROR",
  "message": "Email address is already registered",
  "suggestion": "Please use a different email or try logging in"
}
```

### **5. Ignoring HTTP Semantics**

```http
# ‚ùå Using GET for state changes
GET /api/users/delete/123

# ‚úÖ Use appropriate HTTP methods
DELETE /api/users/123
```

## üöÄ Final Thoughts

Great API design is both an art and a science. It requires careful consideration
of:

- **User Experience** - Intuitive and easy to use
- **Maintainability** - Easy to modify and extend
- **Performance** - Fast and efficient
- **Security** - Safe from attacks and abuse
- **Scalability** - Can handle growth

Remember, APIs are contracts between different systems. Breaking changes can
have cascading effects across multiple applications. Always prioritize backward
compatibility and proper versioning.

**The best APIs are invisible to users but essential to developers.** They work
reliably, perform well, and provide exactly what clients need without
unnecessary complexity.
